# Senra Mobile Redesign ‚Äî Tracked Plan

> **Home:** this doc lives in the **iOS repo** (`pet-safety-ios/`) as the reference platform, but it is **cross-cutting** and governs both `pet-safety-ios` and `pet-safety-android`. Android-specific refs are pathed from `pet-safety-android/`.
> **Status:** Unit 0 complete (investigation + this doc). No feature code written, no branch created.

## 1. Purpose & current status

Restructure the Senra mobile apps so that surfaces which are **already anonymous-capable server-side** (scan ‚Üí public profile, found-stray report, community / lost-&-found reads, pet-friendly-places discovery) become reachable **without logging in**, behind a new logged-out **landing** screen. Fix the guest tag-ordering dead-end (mobile drops the `userId` the checkout needs) and correct inaccurate OTP copy.

**Current status:** Phase 0 (read-only investigation) is **done** ‚Äî findings in ¬ß5, gaps in ¬ß6. Nothing is built. Next unit is Phase 1 (shell + splash + landing), pending review of this doc.

**Working model (two-seat):** CC investigates/builds/surfaces; chat reviews diffs byte-level; **Viktor owns all git**. One reviewed unit per commit. Unit N+1 does not start until unit N is reviewed, tested, and committed.

## 2. Scope

### In scope
- Guest/public pre-login access to surfaces that are **already anonymous-capable server-side**: scan ‚Üí public profile, found-stray report, community / lost-&-found reads, pet-friendly-places discovery.
- New **landing** screen (default logged-out state) + refreshed splash.
- **Guest tag-ordering wiring:** capture `userId`/`email` from `createOrder` and thread into `CreateTagCheckoutRequest` on the **guest path only** (authed order path untouched).
- **OTP copy fix** ("check your email for a code" is inaccurate ‚Äî no code is sent until the user requests OTP at login).

### Out of scope / deferred
- **ANYTHING that reads, writes, or maps into the NAV / Sz√°mla invoicing flow.** Separate, parallel work. Invoicing gaps are **documented here (see ¬ß6), never actioned here.**
- **Separate billing/shipping ADDRESS CAPTURE (Phase 4)** ‚Äî DEFERRED until the invoicing work lands, because its only value is on the invoice, which is across the boundary.
- **B2B invoicing identity** (future `orders.billing_info` column) ‚Äî out.
- Any net-new feature beyond access-restructuring.

### Locked decisions
- **Shell approach = Option A:** add a logged-out-default branch so a new `LandingView` becomes the default unauthenticated state, leaving `MainTabView` / `MainTabScaffold` untouched.
  - iOS: near-drop-in at `ContentView.swift:15‚Äì33`, gated solely on `authViewModel.isAuthenticated`.
  - Android: near-drop-in at `PetSafetyApp.kt:275‚Äì318` ‚Äî a single `screenKey` `when`-block. Add a `"landing"` case; no nav-graph rewrite.
- **Public surfaces gate at the ACTION level, not the screen level:** view public; actions like create-alert / submit-place / activate-tag prompt login at the point of action, reusing the existing deep-link-sheet pattern (iOS `ContentView.swift:53/:84`; Android the existing activation/login-prompt route).
- **Small, independently testable chunks; one surface per chunk in Phase 2.**
- **Logged-out affordances are strictly limited (Q1 ‚Äî decided 2026-07-09):** an unregistered / logged-out finder gets exactly **two** actions and no others ‚Äî (1) **Report a sighting** (the existing sighting flow) and (2) **Scan a tag** (the existing scan flow ‚Üí the found pet's public profile page). No edit, no mark-found, no other owner/account actions are shown to logged-out users. On the missing-pet **alert detail** this means it renders as a public read exposing **only** those two actions; every `currentUser`-gated affordance is hidden when logged out.
- **Landing structure = Shape A, "three-zone community home base" (Q5 ‚Äî decided 2026-07-10):** the default logged-out landing is a three-zone layout designed so future community features **append to an existing structure** rather than trigger a landing redesign:
  - **Zone 1 ‚Äî Act now** (compact action row, top): **Scan a tag** + **I-found-a-stray** ‚Äî prominent, fast, visually tight (the urgent-finder entry). Both trivial/anonymous (¬ß4 2.1 / 2.2).
  - **Zone 2 ‚Äî Order a tag** (distinct product CTA, at the seam): first-class, styled **distinctly** from the acute actions (product/conversion intent, not finder intent). Entry to Phase 3.
  - **Zone 3 ‚Äî Community** (the growable body): a titled section that **renders a list of community entries** ‚Äî seeded with two today (Lost & Found board, Pet-friendly places). Built as a **data-driven rendered collection** (array of entry descriptors ‚Üí cards), **not** two hardcoded tiles. Adding a future community feature = appending an entry, **no landing-layout change**. Board/places reads are anonymous (¬ß4 2.3 / 2.4).
  - **Persistent ‚Äî Sign in / Register:** quiet, secondary, always reachable.
  - Two guardrails are the point of Shape A: **(G-a)** **no empty "coming soon" placeholders** in the community section ‚Äî "room to grow" means the list is *extensible* (structural, invisible until used), presenting as a complete two-entry section, not visible empty slots; **(G-b)** community card/list styling **must reuse existing components** from the ¬ß5.3 inventory ‚Äî Shape A leans harder on shared components than a flat landing, so the Phase-1 spec must **name** which existing components the community cards reuse, and if none fit cleanly that is a finding **CC surfaces before building**, not a new component minted mid-chunk.
- **Android carries a POINTER, not a copy (decided 2026-07-17 ‚Äî extends Q6, does not amend it).** An Android-seat session opened in `pet-safety-android` cannot see the protocol, the plan, the spec, or the board ‚Äî every document governing its work lives in `pet-safety-ios`. **The fix is a pointer, never a mirror.** Q6 ruled out a copy for the plan; the same reasoning binds here ‚Äî a copy drifts from its original within a week, and that drift is the failure the protocol exists to kill. So `pet-safety-android/SENRA.md` (`f0c5aa3`, 2026-07-17) **states no facts**: it points at `pet-safety-ios/docs/{PROTOCOL,SENRA-MOBILE-REDESIGN,phase-1-spec}.md` + `scripts/senra-status.sh`, and says to run the board from the iOS checkout. It is the **HANDOVER pattern applied cross-repo** ‚Äî a file with nothing in it to be wrong about cannot go stale.
  - **The board needed no change and got none.** It already reads both repos ‚Äî `senra-status.sh:17-19` resolves `IOS`/`AND`/`DL` from the environment, so it runs correctly *from the iOS repo against Android today*. There is no Android board and none was minted. The gap was never "the board can't see Android"; it was "an Android seat can't see the docs."
  - **What this forbids:** copying, mirroring, or symlinking any governing doc into `pet-safety-android`. If the pointer ever needs to state a fact to be useful, that is the signal the fact is in the wrong document ‚Äî move the *reader*, not the *text*.

### 5.6 Q2 + Q4 verification (2026-07-10)

**Q2 ‚Äî report-a-sighting is anonymous-capable end-to-end ‚Üí verdict (A): no backend change.** Chain verified backend ‚Üí DB ‚Üí both clients:
- `POST /alerts/:alertId/sightings` (`backend/src/routes/alert.routes.ts:281`) is **`optionalAuth`** (`:282`), not `authenticateToken` (contrast: `/missing`, `/found`, `PUT /:id`, and owner-only `GET`/`DELETE /:id/sightings` are all `authenticateToken`). ### 9.12 üî¥ The false-evidence pattern ‚Äî four for four in one session (2026-07-14)
